# Security Governance — Hylono Stack v5.6

**Last updated:** 2026-02-18  
**Applies to:** All services in the Hylono Stack

---

## 1. Scope

This document defines the security governance framework for the Hylono Stack — a 59-service self-hosted infrastructure platform. It covers access control, secret management, network isolation, audit logging, and vulnerability management.

---

## 2. Secret Management

### 2.1 Principles
- All secrets are stored exclusively in `.env` (never committed to VCS)
- `.env.example` contains only `CHANGE_ME_*` placeholders
- Secrets are generated with cryptographically secure random sources (`openssl rand` or equivalent)
- All secrets are unique per deployment — no shared values across services

### 2.2 Secret Generation
Secrets are generated by `docker/infrastructure/setup.sh` using a while-loop deduplication pattern:
```bash
# Each secret is checked for uniqueness before assignment
while IFS= read -r line; do
  key=$(echo "$line" | cut -d= -f1)
  secret=$(openssl rand -hex 32)
  echo "${key}=${secret}"
done < .env.example > .env
```

### 2.3 Required Secrets
| Variable | Length | Purpose |
|----------|--------|---------|
| `POSTGRES_ROOT_PASSWORD` | 32 bytes hex | PostgreSQL superuser |
| `REDIS_PASSWORD` | 32 bytes hex | Redis auth |
| `MINIO_ROOT_PASSWORD` | 32 bytes hex | MinIO admin |
| `ZITADEL_MASTERKEY` | 32 bytes hex | Zitadel encryption |
| `SECRET_KEY_BASE` | 128 bytes hex | Rails app base secret |
| `*_JWT_SECRET` | 64 bytes hex | JWT signing keys |
| `*_ENCRYPTION_KEY` | 32–64 bytes hex | At-rest encryption |

## 3. Network Isolation

### 3.1 Docker Network Segmentation
- All services run on `hylono-internal` bridge network (no public exposure)
- Only Kong (port 8443) and Uptime Kuma (port 3001) are exposed to the host
- Inter-service communication uses Docker internal DNS (`service-name:port`)

### 3.2 Port Policy
- No service exposes ports externally except via Kong API Gateway
- Kong enforces TLS termination and rate limiting
- Direct service port access is disabled in production

---

## 4. Authentication & Authorization

### 4.1 Identity Provider
- **Zitadel** is the primary IdP for all Hylono services
- OAuth2 / OIDC with PKCE required for all user-facing services
- Service-to-service: API keys via Infisical vault

### 4.2 Access Levels
| Role | Access |
|------|--------|
| Admin | Full stack control panel + all services |
| Operator | Service management, no credential access |
| Viewer | Read-only dashboards (Metabase, Grafana) |
| Service Account | API key scoped to single service |

---

## 5. Audit Logging

- **Retraced** provides immutable audit trails for all admin actions
- **Wazuh** provides SIEM, intrusion detection, and file integrity monitoring
- All authentication events logged to Zitadel audit log
- Kong request logs forwarded to Wazuh via syslog

---

## 6. Vulnerability Management

### 6.1 Dependency Scanning
- Docker images pinned to specific digest hashes in production
- Weekly automated `docker pull` + digest comparison
- CVE scanning via Wazuh SIEM alerts

### 6.2 Patch Policy
- Critical CVEs: patched within 48 hours
- High CVEs: patched within 7 days
- Medium/Low: batched monthly

---

## 7. Compliance

### 7.1 GDPR
- Data processing agreements in place for all services handling EU personal data
- Data minimization enforced at the Kong gateway level
- Right-to-erasure workflow documented in `data-governance.md`

### 7.2 Data Residency
- All services deployed in EU data centers (Frankfurt / Warsaw)
- No data transfer outside EU without explicit consent

---

## 8. Incident Response

1. **Detect** — Wazuh alert or manual discovery
2. **Contain** — Isolate affected service (`docker stop <service>`)
3. **Investigate** — Review Retraced audit log + Wazuh timeline
4. **Remediate** — Rotate secrets, patch image, redeploy
5. **Post-mortem** — Document in `.agent/memory/active/errors.md`
